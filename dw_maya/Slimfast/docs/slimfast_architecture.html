1. What Slimfast is, in one minute

Slimfast is a Maya weight-painting tool. A character mesh can carry many different kinds of per-vertex weight: a skinCluster has one weight per bone, a blendShape has base weights, an nCloth has dozens of named simulation maps (thickness, stretch…), and so on. Historically each of these needed a different tool, different buttons, and different Maya commands.

Slimfast hides all of that behind one interface. You select a mesh, Slimfast lists every paintable weight source on it, and the same buttons — flood, smooth, mirror, select-by-range, copy/paste, paint — work on all of them. The UI never asks "is this a skinCluster or an nCloth?"; it just calls the same handful of methods on whatever weight source is active.

Making "the same buttons work on everything" possible is what most of this document is about. Three ideas do the heavy lifting:

  1. Node wrappers (MayaNode) — a friendlier handle on a Maya node.
  2. Factories & registries — code that hands you the right wrapper for a node without you writing if/elif chains.
  3. A shared contract (WeightSource) — every weight source implements the same small set of methods, so callers stay type-blind.

2. Glossary

TermMeaning
DeformerA Maya node that changes a mesh's shape: cluster, blendShape, skinCluster, wire, softMod…
Influence / boneA joint that a skinCluster is bound to. Each vertex has one weight per influence; the weights of a vertex sum to 1.0.
Weight mapOne float per vertex (0..1 usually). A skinCluster has one map per influence; an nCloth has many named maps.
MObjectOpenMaya's internal handle to a node. Survives renaming (unlike a name string).
FactoryA function whose job is to build and return the right object for you (see §5).
RegistryA lookup table mapping a key (e.g. a node type) to a class. The factory reads it.
Protocol / ABCA promised set of methods a class agrees to provide. Code can rely on the methods without knowing the concrete class. WeightSource is ours.
ArtisanMaya's interactive paint tool (the brush). Slimfast can drive it or bypass it with direct numpy writes.
ControllerThe class that holds all Maya logic for Slimfast (SlimfastController). The UI never calls Maya directly.

4. The node wrappers — MayaNode

Lives in dw_maya/dw_maya_nodes/. These wrappers give you PyMEL-style ergonomics (dot access to attributes, connections, presets) while keeping cmds speed underneath. Three classes:

ClassRole
ObjPointerBase class. Stores an MObject/UUID so the wrapper keeps working after the node is renamed.
MayaNodeHigh-level node wrapper (inherits ObjPointer). Attribute access, transform/shape duality, presets.
MAttrSingle-attribute wrapper returned by attribute access — get/set, connect (>>), compare, slice.
cube = MayaNode('pCube1')
cmds.rename('pCube1', 'myCube')
print(cube.node)            # 'myCube'  — pointer still valid after rename

cube.translateX            # an MAttr
cube.translateX.get()      # read
cube.translateX.set(10)    # write
cube.translateX >> other.translateX   # connectAttr

MayaNode('myLocator', 'locator')       # second arg is a type -> creates the node
Why MObject and not the name string? Names change (rename, namespace import, reference). An MObject is a stable handle, so a wrapper you stored five operations ago still points at the right node. This is the same reason the skinning code resolves influences by MObject/index, not by name.

5. Factories & registries — the core idea

5.1 What is a factory? (no Maya)

A factory is just a function whose job is to decide which class to build and return it to you, so the caller doesn't have to. Naïve code without a factory grows an ever-longer if/elif chain that every caller must repeat:

# WITHOUT a factory — every caller duplicates this and must be edited
# whenever a new shape is added.
def make_shape(kind):
    if kind == 'circle':
        return Circle()
    elif kind == 'square':
        return Square()
    elif kind == 'triangle':
        return Triangle()
    ...

A registry turns that chain into a lookup table, and the factory reads the table. Adding a new shape becomes one line — and no caller changes:

# WITH a registry + factory
_SHAPES = {}                       # the registry (a dict)

def register(kind, cls):
    _SHAPES[kind] = cls

def make_shape(kind):              # the factory
    return _SHAPES.get(kind, DefaultShape)()

register('circle', Circle)         # a new shape = one register() call
register('square', Square)
Why we use this everywhere. New deformer types, new nucleus maps and new UI panels all get added by registering a class, never by editing a big conditional in the middle of the tool. That is exactly what makes Slimfast safe for you to extend while the author is away: you add a file and one registration line; you don't touch the dispatch logic.

5.2 Our deformer factory — make_deformer()

File: dw_maya/dw_deformers/dw_deformer_class.py. The registry is a plain dict; the factory looks up the Maya node type and falls back to the base Deformer:

_DEFORMER_CLASSES = {
    'cluster':     Cluster,
    'softMod':     SoftMod,
    'blendShape':  BlendShape,
    'wire':        Wire,
    'skinCluster': SkinCluster,
}

def make_deformer(node, preset=None, blend_value=1.0):
    node_type = cmds.nodeType(node)
    cls = _DEFORMER_CLASSES.get(node_type, Deformer)   # <-- the lookup
    ...
    return cls(node, preset, blend_value)
Gotcha worth knowing. There are historically two SkinCluster classes — a light stub and the full implementation in dw_skinning_class.py. The full one re-claims the 'skinCluster' slot in _DEFORMER_CLASSES at import time, so make_deformer('skinCluster1') returns the rich class. If you ever importlib.reload(dw_deformer_class) in isolation you reset the dict back to the stub — reload dw_skinning_class too (or just restart Maya). Sanity check: type(make_deformer(node)).__module__ should end in dw_skinning_class.

5.3 The type registry — lsNode() / dw_node_registry

File: dw_maya/dw_node_registry.py. This is a richer registry used to wrap any node (not just deformers). It supports two kinds of registration — by exact type, and by a condition function — and resolves in a fixed priority order:

# by exact type

# by condition — e.g. "a mesh that is connected to an nCloth node"
def _is_cloth(node):
    return node.type == 'mesh' and bool(cmds.listConnections(node, type='nCloth'))
register_condition(_is_cloth, NClothMap)
Two registries, on purpose. make_deformer uses its own _DEFORMER_CLASSES dict (deformers only, exact-type). lsNode uses dw_node_registry (everything, with inheritance + conditions). They are independent; a class can live in both.

6. The shared contract — WeightSource

File: dw_maya/dw_paint/protocol.py. This is the abstraction that lets the UI stay type-blind. Every paintable thing — a deformer, an nCloth map, a vertex-colour channel — is a WeightSource and provides the same small API:

MemberWhat it does
available_maps()Names of the maps this node exposes (e.g. ['weightList'], or every influence of a skin, or ['thickness','stretch',…]).
use_map(name)Activate one map. Chainable. Single-map nodes auto-resolve so you can skip it.
get_weights() / set_weights(list)Read / write the active map (one float per vertex).
paint()Open the right Artisan tool for the active map.
node_name / mesh_name / vtx_countIdentity helpers.

Because they all share the contract, the entry-point factory returns them in one flat list with no type checks needed by the caller:

from dw_maya.dw_paint.weight_source import resolve_weight_sources, apply_operation

sources = resolve_weight_sources('pSphere1', mode='all')   # deformers + nucleus + vtxColor
src = sources[0]
src.use_map('weightList').get_weights()
apply_operation(src, 'smooth', iterations=3)   # works on ANY source
This is the payoff of §5 + §6. resolve_weight_sources is a factory that asks each backend "what's on this mesh?" and returns a list of WeightSource objects. apply_operation then dispatches a named op (flood/smooth/mirror/vector/radial) to that source. The UI calls these two functions and never branches on node type.

7. Slimfast's own architecture

Slimfast follows a strict UI ↔ Controller split. The UI builds widgets and connects signals; it never imports cmds. All Maya logic lives in the controller. They communicate through a tiny signals object.

7.2 The panel registry

Each deformer type can have its own sub-panel (the skinCluster influence list, the nCloth map-type badge…). Panels register themselves with @panel_for — the same registry pattern as §5. main_ui builds the radio buttons from get_mode_registry() and picks a panel for the active node with get_panel_class(node_type).

8. End-to-end flows

Tracing a single action top-to-bottom is the fastest way to learn the codebase — and to debug it. Two flows cover almost everything: picking a source (which also swaps the panel) and running an operation (clicking "Set 0").

8.1 Picking a source → swapping the panel

Before any button does anything, a WeightSource must be active. That happens in two steps: refresh() reads the Maya selection and fills the source combo; choosing a row activates that source and selects the matching sub-panel. Selecting a row is the "item pick" — it decodes (source_idx, map_name) from the row, activates both on the controller, then asks the panel registry which panel that node type wants.

Two signals, two jobs. sources_changed rebuilds the list of rows; active_changed fires whenever the active source/map changes and is what every panel and the shared buttons (Set 0/1, paint, copy/paste) listen to via on_source_changed. The panel swap is driven off the row's stored node_type (UserRole+1), not off the signal — so the registry (§7.2) decides which panel a node type gets, exactly like §5.

Debugging mindset. If "Set 0" misbehaves, walk this chain top-down: is the button wired (main_ui)? Does the controller get the right value and selection (cmds.py set_weight)? Does the source do the right math (flood/_lock_aware_write)? Did the write land in Maya (turn on DEBUG logging — the writers log per-pass results)? Each layer is small and testable on its own.

9. How to add a new deformer panel

Say you want a custom panel for wire deformers. You add one file and one decorator — no existing file changes.

# dw_maya/Slimfast/wgt_wire_panel.py
from dw_maya.Slimfast.wgt_deformer_panel import DeformerPanelBase, panel_for

@panel_for(
    node_types = ['wire'],     # which Maya types trigger this panel
    label      = 'Wire',       # radio-button text
    ctrl_mode  = 'deformer',   # which controller mode to use
    order      = 18,           # left-to-right order of the radio button
)
class WirePanel(DeformerPanelBase):
    def build_body(self):
        self._lbl = QtWidgets.QLabel('Wire deformer')
        return self._lbl

    def on_source_changed(self, source, active_map, ctrl):
        self._lbl.setText(source.node_name if source else '—')

Then import the module once (so the decorator runs) — the same way wgt_skin_panel is imported in main_ui.py. That's it: the radio button appears, and the panel shows whenever a wire is the active source.

OverrideWhen

Pattern reuse. wgt_skin_panel.py is the worked "complex panel" example — a model/view tree with a custom lock column and a prune field. Copy its shape when your panel needs real interaction.

10. How to add a new advanced operation (mirror-style)

Advanced ops (the existing vector and radial distributions, and mirror/smooth) follow a consistent four-layer recipe. To add one — call it gradient — touch these four places:

  1. Pure function in dw_maya/dw_paint/operations.py that takes a mesh + parameters and returns a new per-vertex weight list. Keep it Maya-light and side-effect-free so it is unit-testable. 
    def set_gradient_weights(mesh, ..., invert=False):
    # compute and return List[float], one per vertex
    return new_weights
  2. Dispatch: expose it through apply_operation (the op map in weight_source.py) or, for ops that need bespoke selection/centre handling, add a dedicated controller method (mirror what apply_vector_weights / apply_radial_weights already do).
  3. Controller method in cmds.py, decorated with @singleUndoChunk (one undo step) and respecting the active mask / selection. Read with self._active.get_weights(), combine with the op result per the chosen op (replace/add/subtract/multiply), and write back with self._active.set_weights(result). 
    @singleUndoChunk
def apply_gradient_weights(self, ..., op='replace'):
    if not self._require_active():
        return
    new_weights = set_gradient_weights(self._active.mesh_name, ...)
    weights = list(self._active.get_weights())
    mask = self._mask_vtx_ids        # advanced-ops vertex mask, or None = whole mesh
    ...combine + write...
    self._clamp_weights_post(mask)   # honour the clamp checkboxes
  4. UI in main_ui.py: add the widgets (button + any params) and connect the button to the new controller method. The UI never touches Maya — it only calls self._ctrl.apply_gradient_weights(...).
Two rules that keep ops correct. (1) Operations are selection-aware: respect _mask_vtx_ids / the live vertex selection so a partial paint stays partial. (2) For skinCluster, writing the active influence is not a plain column write — set_weights routes through the lock-aware redistribution (_lock_aware_write) so locked bones are never disturbed and per-vertex totals stay 1.0. You get that for free by going through set_weights; do not call the API directly.

11. Debugging checklist

SymptomFirst thing to check
"My edit to a class did nothing."Reload the toolkit: from dw_maya import reload_dw_toolkit; reload_dw_toolkit(). For skinCluster edits, confirm make_deformer returns the rich class (§5.2 gotcha).

12. Conventions & gotchas

  • Imports: import module then module.func(); avoid import * and import x as y except where noted.
  • Naming: classes & tool packages CamelCase; functions lower_case; inline type hints.
  • Python 3.7 target: no walrus :=; f-strings for formatting.
  • PySide2 / PySide6: production Maya may ship only PySide2. New code prefers PySide6 but always provides the PySide2 fallback (the try/except import block at the top of each UI file). Watch enum-based widget APIs and exec_() → exec(). Always assign new QStandardItem/Qt objects to a variable (Python/C++ garbage-collection mismatch), and pair every model with a proxy model.
  • OpenMaya API 1.0 vs 2.0: for multi-influence skin writes the API 2.0 MFnSkinCluster.setWeights mis-dispatches; the centralized writer write_influence_columns uses API 1.0. Don't reintroduce direct API 2.0 multi-influence writes.
  • Undo: wrap user-facing controller ops in @singleUndoChunk so one click = one undo step.
